# Бизнес-анализ проекта — сервис приоритизации инфоповодов (НТС)

## 1. Бизнес-анализ (Business Understanding)

### Цели и скоуп проекта

**Организационная структура:** со стороны заказчика участвует директор и мой заместитель. Основной пользователь на первом этапе — ООО «НТС», далее это решение можно продавать с адаптацией редакциям в других регионах (предварительный интерес есть).

**Главная бизнес-цель:** увеличить просмотры на сайте телеканала — в условиях ограниченных ресурсов (количество сотрудников, время на обработку новости) выбирать инфоповоды, которые с большей вероятностью наберут больше просмотров.

**Существуют ли какие-то уже разработанные решения:** есть несколько кастомных внутренних решений у нескольких редакций, но другим редакциям нельзя там стать пользователями. Из публичных — [givemepublic.com](https://givemepublic.com/). Минус: по факту путается в датах и может собрать несвежие новости и выдать их как свежие, низкий процент уникальности сгенерированных рерайтов, отсутствие возможности сделать тонкие настройки/подстройки в качестве рерайтов.

## 1.1 Текущая ситуация (Assessing current solution)

### Наличие ресурсов для проекта

**Доступность железа:** в наличии CPU-сервер редакции, пригодный для инференса лёгких моделей и пайплайна сбора/обработки данных. Собственного GPU нет; обучение и дообучение моделей планируется выполнять на доступных GPU (Colab / Kaggle), а также на арендованных мощностях в рамках ограниченного бюджета.

**Хранение и доступность данных:**
- *Внешние открытые источники* — RSS-ленты региональных и федеральных СМИ, Telegram-каналы, новостные агрегаторы и публичные датасеты.
- *Внутренний архив новостей телекомпании* — опубликованные материалы в CMS Битрикс, метрики просмотров из Я.Метрики, а также аналитика собственных каналов в соцсетях.

**Эксперты заказчика для консультации:**
- директор — валидация бизнес-ценности и приоритетов;
- редактор/SMM-специалист — разметка «важности» инфоповодов и экспертная оценка качества рерайтов.

Консультации — на регулярной основе, т.к. это наша внутренняя команда.

### Риски и митигация

| Риск | Проявление | Митигация |
|---|---|---|
| Не уложиться в сроки | Тяжёлое обучение моделей, объёмный пайплайн | MVP-first подход: разбить проект на обязательное ядро и необязательные надстройки — тогда даже если времени не хватит на всё желаемое, на руках будет законченный рабочий продукт |
| Результат не интересен заказчику | Рерайты не дают эффекта | Итеративный фидбек по улучшению, отслеживание ключевых показателей, привязка к измеримому эффекту |
| Галлюцинации / фактические ошибки в рерайте | Модель искажает факты | Human-in-the-loop: рерайт = черновик с подсветкой источника; факт-сверка редактором перед публикацией |
| Низкая уникальность рерайта | Низкий процент по проверкам text.ru | Контроль уникальности за счёт разбиения n-gramm, работы с шинглами и эмбеддингами. Проверенный справочник для синонимизации |
| «Несвежие» новости / ошибки по гео новости | Выдача устаревшего за новое / выдача новостей других регионов | Отдельный модуль детекции свежести/даты события; фильтр по временному окну и географии |
| Неполные данные для рерайта | Есть опубликованные тексты, но мало пар «источник → рерайт» | Дообучение под стиль на одних опубликованных текстах (адаптация стиля), а исходник подаётся в запрос; небольшой набор пар собирается полуавтоматически для оценки |

## 1.2 Решаемые задачи с точки зрения аналитики (Data Mining goals)

**Главная метрика.** По сути, главная бизнес-цель — это околобинарная задача: решать публиковать или не публиковать инфоповод (их ориентировочно от 15 до 80–120 в день в регионе). Поэтому в качестве метрики берём **ROC-AUC**. Целевая переменная — бинарная: «залетит» (1), если статья набрала **≥ 90 просмотров**, иначе 0. Порог обоснован распределением просмотров на статью: по очищенному от архива датасету (статьи сопоставлены с картой сайта по идентификатору) медиана составляет 47 просмотров, и порог 90 отделяет верхние ~20% материалов — то есть явно «залетевшие» новости от фоновых. Это отражает бизнес-цель — отбор инфоповодов с высоким потенциалом просмотров при ограниченных ресурсах редакции.

**Вспомогательная метрика 1 — фактологическая точность рерайта.** Эта метрика играет роль «предохранителя»: рерайт не должен искажать факты исходной новости, иначе возникает репутационный и юридический риск. Современная практика (2024–2025) отказалась от оценки рерайта только по ROUGE в пользу метрик согласованности с источником — использую **AlignScore** либо **SummaC**, дополняя оценкой LLM-as-judge и оценкой человека-редактора. Критерий успеха здесь не пороговое число, а отсутствие фактических ошибок на экспертно проверяемой выборке.

**Вспомогательная метрика 2 — фильтр релевантности (свежесть + гео)** — Precision и Recall. Высокий Precision здесь принципиален: он закрывает главную слабость существующих решений вроде givemepublic, которые выдают устаревшие новости как свежие. Целевой Precision ≥ 0.90.

**Вспомогательная метрика 3 — уникальность текста**, считается относительно рерайтов этой же новости в других источниках. Анализ идёт через API Text.ru. Пороговое значение — 75%.

## 1.3 План проекта (Project Plan)

Проект разрабатывается итеративно по шести фазам процесса (на основе CRISP-DM) и состоит из трёх подсистем с явными приоритетами: **A. Приоритизатор «залетит / не залетит»** (ядро, оценивается), **B. Отслеживание новостей по лентам** (входная обвязка), **C. Рерайт-ассистент** (расширение: дообучение под tone of voice и правила редакции + фактчек как сверка с источником). Бизнес-часть раскрывается на Уроках 1–3. Моделирование стартует раньше лекции L6 (в конце июня, параллельно) — чтобы к лекциям L5–L6 уже был baseline и конкретные вопросы.
