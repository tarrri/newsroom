# ДЗ4 (модуль данных) — сопроводительная записка

## 1. Про проект

Редакция получает за день десятки инфоповодов при ограниченном ресурсе по их обработке и подготовки к публикации. Сервис предсказывает, **насколько инфоповод залетит** (наберёт просмотров), и ранжирует входящие — чтобы браться за потенциальные хиты. Компоненты: приоритизатор, мониторинг RSS-лент (в дальнейшем планирую подключить соцсети), рерайт под Tone of Voice и стиль редакции, фактчерек (сверка рерайта и оригинала)

---

## 2. Источник и состав данных

- **Просмотры** — Яндекс.Метрика, через API выгруженя данные по текстам за 7,5 лет (ноя.2018–июн.2026), просмотры по URL по дням.
- **Тексты статей** — спарсены со страниц nts-tv.com.
- Итог: **~13 000 статей, 52 месяца**, 3 класса по просмотрам (низкие 6491 / средние 3936 /
  высокие 2642). Просмотры в признаки не подаются, чтобы избежать утечки, они используются только для присвоения правильной метки.

---

## 3. Качество данных во времени

- **Тренд/дрейф:** трафик рос к 2020 (пик COVID 255k/мес) → спад в ~3–4 раза к 2024.
  Этот момент снят **относительным таргетом внутри месяца** (абсолютный уровень не искажает метку).
- **Сезонность — практически нет:** сезонный индекс 0.76–1.06, автокорреляция
  лаг-12 ≈ −0.06. Трафик гонят **события**, не смена сезонов.
- **Датирование** — по реальной дате публикации (отсекает всплывший старый архив).

---

## 4. Качество разметки и предложения

Таргет (метка из просмотров) строится автоматически. Вспомогательная LLM-разметка типов/флагов **валидирована редактором** на стратифицированной выборке 100 статей:
- согласие по **типу события 94%**;
- **по флагам 97%**, почти без возражений.

**Предложение по повышению качества разметки:** сделать разметку **мультиметочной** (основной + дополнительный тип). Для модели не критично — предиктор работает по тексту, теги нужны лишь для интерпретации.

---

## 5. Как консультация повлияла на работу

После консультации я внесла ряд изменений. Главное — перешла от бинарного таргета к трёхклассовому (низкая  средняя / высокая) и обосновала порог lift-анализом. Закрепила валидацию по времени (out-of-time, скользящее окно). Добавила роутинг по «живучести» новости (разовая / сезонная / нетленка) и фильтр микро-одноразовых событий, которые почти всегда не залетают. Также начала работать над ранней классификации. Бейзлайном взяла TF-IDF + логистическую регрессию. Проверила сезонность автокорреляцией: на уровне трафика её нет, доминируют тренд и события — это стало отдельной находкой. С учётом дисбаланса классов сделала главной метрикой PR-AUC вместо ROC-AUC.

## 6. Результаты (выводы для моделирования)

Валидация — out-of-time скользящее окно по месяцам. Провела быстрое первоначальное сравнение признаков \ нескольких моделей.

| признаки / модель | ROC-AUC |
|---|---|
| структурные (event_type + флаги) | 0.615 |
| **TF-IDF текста + LogReg** | **0.669** ← лучший |
| бустинг на сыром тексте | 0.545 (переобучается) |

**Метрики 3 классов:** 
- **главная PR-AUC = 0.405**
Классы несбалансированы: хитов всего ~20% (1 к 4), а при таком дисбалансе ROC-AUC рисует слишком оптимистичную картину — поэтому главной взята PR-AUC. Её читают относительно доли класса: у случайной модели PR-AUC равна базовой доле хитов, то есть 0.20; у нашей — 0.405, вдвое выше случайного.

- **операционная Precision@top-20% = 0.36**
Если редакция берёт топ-20% инфоповодов по оценке модели, среди них реальных хитов 36% против 20% при случайном выборе — то есть в 1.8 раза больше попаданий на тот же объём работы.

- **ординальная QWK 0.258**
Показывает, насколько прогноз совпадает с фактическим порядком классов (низкая → средняя → высокая): 0 — как случайно, 1 — идеально; 0.26 — умеренное согласие.

ROC-AUC = 0.675 привожу для сравнения: при дисбалансе он завышает оценку, поэтому главной метрикой не берём.

**Ablation глубины окна:** вся история (50 мес) даёт лучший результат — расширение данных помогает, дрейф не вредит → обучаем на всём. Модель у **потолка текстовых признаков ~0.67**; дальнейший рост предполагается за счет работой над датасетом и моделью, не на увеличении объема.

## 7. Алгоритм формирования выборки и пайплайн (воспроизводимость)

Весь путь данных построен на скриптах, датасет пересобирается одной командой:

| Скрипт | Что делает |
|---|---|
| `fetch_metrika.py` | тянет просмотры по статьям из Метрики (API), по дням |
| `make_scrape_list.py` | формирует список URL для докачки из sitemap (минус уже скачанное) |
| `scrape_bodies.py` | качает тексты статей со страниц (обход ddos-guard), возобновляемый |
| `rebuild_dataset.py` | сшивает метрику + тексты, датирует по дате публикации, строит 3-классовый таргет внутри месяца, чистит (отвал/итоги/нулевые просмотры) |

**Стратегия валидации:** учим на ранних месяцах, тестим на поздних (как на проде), и усредняем по нескольким месяцам. Утечки под контролем: убрали дайджесты-«итоги» (пересказ вышедшего); настоящих дублей почти нет, а те, что есть, лежат внутри одного месяца — разбиение по времени их разделяет.

---

## 8. Дополнительные аспекты

- **Воспроизводимость/версионирование:** пайплайн сделан из скриптов; формирования датасета из информации, собранной из нескольких источников = одна команда.
- **Смещение выборки (bias):** модель учится только на **опубликованных** статьях (которые редактор уже отобрал) — это selection bias относительно входящего потока на проде. Гео — константа (Севастополь).
